# 流程 C：编译、质检与导出

流程 C 在流程 B 通过完成门禁后启动，负责把已写入当前模板根目录的 LaTeX 工程编译成 PDF，并完成诊断、有限机械修复、质检和交付说明。

流程 C 不重新判断论文内容归属，不修正文档语义。章节归属、摘要关键词、图表归属、参考文献内容真实性、正文删改等内容问题必须回到流程 B。

## C.1 边界

流程 C 的职责：

- 归档或清理旧编译产物，避免误判旧 PDF。
- 运行固定 LaTeX 编译链。
- 收集编译日志和退出码。
- 诊断编译和质检问题。
- 自动修复有限的机械错误。
- 生成新的 `main.pdf`。
- 生成报告和质检结果。
- 给用户简短交付说明。

流程 C 不负责：

- 重新组织论文章节。
- 改写正文含义。
- 判断摘要、关键词、图表或参考文献应该归属于哪里。
- 编造或修正参考文献真实性。

## C.2 编译前旧产物处理

流程 C 开始前必须处理旧产物，避免旧 `main.pdf` 被误判为本轮结果。

旧 `main.pdf` 归档到：

```text
workspace/archive/<timestamp>/flow-c-before-build/main.pdf
```

旧临时编译文件应清理或归档，例如：

- `main.aux`
- `main.bbl`
- `main.bcf`
- `main.blg`
- `main.log`
- `main.out`
- `main.run.xml`
- `main.toc`

流程 C 必须记录本轮编译开始时间，并确认最终 `main.pdf` 是本轮新生成的文件。

## C.3 编译链

固定编译链：

```text
xelatex main.tex
biber main
xelatex main.tex
xelatex main.tex
```

每一步都要记录：

- 命令。
- 退出码。
- 开始和结束时间。
- 关键日志路径。
- 是否生成或更新目标产物。

如果任一步失败，流程 C 进入诊断和有限修复流程。

## C.4 有限机械修复

流程 C 允许自动修复不涉及论文语义的机械错误：

- LaTeX 特殊字符转义问题。
- 图片路径或大小写错误。
- 缺失资源目录。
- 明显引用键格式问题。
- 临时文件污染。
- 可由日志明确定位的资源路径问题。

流程 C 不允许自动修复内容语义：

- 正文属于哪一章或哪一节。
- 摘要和关键词的内容取舍。
- 图表应该放在哪里。
- 参考文献条目是否真实或应该如何补全。
- 正文段落删改。

自动修复和重新编译最多 2 轮。超过 2 轮仍失败时，流程 C 停止并输出分类诊断。

## C.5 失败分类

编译或质检失败时，流程 C 必须分类，而不是直接把 LaTeX 报错丢给用户。

分类至少包括：

- `mechanical_fixable`：机械错误，C 可以自动修复并重试。
- `return_to_flow_b`：内容结构或语义问题，需要回到流程 B。
- `user_input_required`：缺少用户必须提供的文件、图片、文献或确认。
- `environment_issue`：`xelatex`、`biber`、字体、包或系统环境问题。
- `unclassified_failure`：暂时无法分类，需要保留日志并说明人工排查入口。

每个失败项都要给出下一步，而不是只给错误日志。

## C.6 质检范围

第一版质检覆盖日志、文本和页级检查，不做 PDF 截图视觉检查。

质检项包括：

- 本轮是否生成新的 `main.pdf`。
- PDF 页数是否大于 0。
- PDF 文本是否能被读取。
- 编译日志是否存在严重错误。
- 是否存在未解析引用或问号引用。
- 参考文献是否输出。
- 目录、摘要、正文、参考文献等关键章节信号是否存在。
- 是否残留模板蓝字或模板说明。
- 是否残留明显占位符，例如 `xx`、`xxxxxxxxxxxx`、`本文……`。
- 输出报告是否生成。

PDF 生成但存在模板蓝字、占位符、可疑文本或非阻断风险时，最终状态不能是 `ready_to_submit`。

## C.7 输出位置

最终 PDF 保留在项目根目录：

```text
main.pdf
```

报告类文件放入：

```text
workspace/output/report.md
workspace/output/qa_report.md
```

`report.md` 面向 Codex 和技术用户，记录编译、修复、失败原因和日志位置。

`qa_report.md` 记录质检项、状态、证据和最终交付判定。

## C.8 最终状态

流程 C 最终状态分为三类：

- `ready_to_submit`：PDF 生成且质检通过。
- `needs_review`：PDF 生成，但存在模板蓝字、占位符、可疑文本或非阻断风险。
- `failed`：无新 PDF、编译失败、关键页面缺失、引用或资源严重错误，或需要回到流程 B 修复。

成功反馈面向非技术用户，只给简短交付说明：

- PDF 文件位置。
- 最终状态。
- 是否建议提交。
- 报告位置。
- 必须处理的问题。

## C.9 scripts 与 Codex 分工

scripts 负责稳定编译、日志解析和质检：

- `build.py`：归档旧产物、运行编译链、记录命令、退出码、日志路径和是否生成新 PDF。
- `diagnose_build.py`：解析 `main.log`、`main.blg` 等日志，把问题分类为机械可修、需回 B、需用户补充或环境问题。
- `qa.py`：检查 PDF 页数/文本、关键章节信号、未解析引用、模板蓝字、占位符和输出完整性。

Codex 负责解释分类结果、决定是否执行机械修复、向用户说明风险，并在需要时引导回流程 B。

## C.10 测试场景

- 无旧产物时：编译成功后根目录生成新的 `main.pdf`，报告进入 `workspace/output/`。
- 有旧 `main.pdf` 时：旧 PDF 先归档，编译失败不得误判成功。
- 缺图片路径：流程 C 自动修复路径或资源目录问题，最多重试 2 轮。
- LaTeX 特殊字符错误：流程 C 可做机械转义修复并重编译。
- 章节归属或正文内容问题：流程 C 不修，分类为需回流程 B。
- PDF 生成但仍有 `xx`、`xxxxxxxxxxxx`、`本文……`、蓝字模板说明：状态为 `needs_review`。
- PDF 生成且无关键风险：状态为 `ready_to_submit`。
- 编译失败或无新 PDF：状态为 `failed`，输出分类诊断和下一步。
